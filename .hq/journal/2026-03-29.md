## Decisions
- **Scoring curve k=12 + hue recovery**: k=20 too generous, k=12 too harsh for same-hue misses. Hue-recovery bonus (40% at ≤25°) bridges the gap.
- **No SPA rewrite on Vercel**: catch-all rewrite conflicts with framework detection. FastAPI redirect handles root URL.
- **Reveal screen: score + verdict only**: ΔE is jargon. Score colored to match target.
- **HSB slider visualization over delta bars**: players think in HSB, not CIELAB.
- **Picture It click-to-submit**: the choice IS the confirmation.
- **Don't split the file yet**: 2500 lines but no forcing function.
- **Frontend overhaul is staged, not monolithic**: decomposed across alpha (error/loading states) and beta (design pass, icons, animation).
- **Milestones are capability boundaries, not backlogs**: short phrase for remaining work, detailed items live in specs-roadmap.
- **Client-side color generation**: server did nothing but `random()` + perceptual distance checks. Moving it client-side eliminates cold start latency and makes the game work offline. Server-side generation kept as dead code for Phase 3 (seeded challenges).
- **Picture It target is HSB text, not a swatch**: the text IS the game mechanic — Decoding skill means reading numbers and mentally constructing the color. Showing a swatch would reduce it to trivial matching. Initially misidentified as a bug ("HSB leak"), reverted after checking `identity-model`.
- **Analytics must never crash the game**: no-op stub guard before main script ensures all `analytics.*` calls are safe even if `analytics.js` fails to load (ad blocker, CDN issue, dev server).

## Vision
- **The game teaches through color, not about color.** Remove numbers, replace with visual comparisons. Players should be seeing their mistakes, not reading them.
- **Hue is the primary skill axis.** Chromatic discrimination is the hardest perceptual skill. Scoring rewards it over brightness/saturation matching.
- **Per-color reveal changes the learning model.** Immediate reinforcement for skill acquisition, not delayed batch assessment.
- **Infrastructure should be invisible to training.** Client-side generation isn't a Vercel workaround — training tools should never break, never wait for a network.
- **Each mode's UI serves its skill.** Picture It shows numbers because Decoding IS reading numbers. "Visual over textual" applies to feedback, not to game mechanics that test textual-to-visual translation.

## Direction

**Durable:**
- Hue accuracy matters more than brightness/saturation drift — reward the right color family
- Visual over textual for *feedback* — sliders with pins, not HSB numbers or CIELAB deltas
- Mode UI serves the skill being trained — don't apply blanket rules across modes
- Strip containers that only frame other elements
- Milestones track capability inflection points, not task lists

**Scaffold** (model failure mode):
- Curve and tier thresholds are coupled — don't adjust one without the other
- Don't conflate visual polish with resilience — they have different release gates
- Always browser-test before deploying — syntax checks and curl are not enough

**Active tuning:**
- Scoring: k=12 + hue recovery (40% at ≤25°). Tiers: 42/35/25/15. Expect further iteration from playtesting.
